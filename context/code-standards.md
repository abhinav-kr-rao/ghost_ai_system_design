# Code Standards

## General

- Keep modules small and single-purpose.
- Fix root causes — do not layer workarounds.
- Do not mix unrelated concerns in one component or route.
- Respect the system boundaries defined in `architecture-v2.md` and the document-based access model defined in `project-overview-v2.md`.

## TypeScript

- Strict mode is required throughout the project.
- Avoid `any` entirely. This is strictly enforced for Liveblocks storage payloads and Prisma queries.
- Use Zod schemas to validate all external inputs. This is especially critical for validating the structured JSON output generated by the Vercel AI SDK before it touches the canvas.
- Use explicit `interface` declarations for object contracts, particularly for defining custom node properties within the Liveblocks room state.

## Framework Patterns (Next.js & Real-time)

- Default to React Server Components for all standard page rendering and layout construction.
- Add `"use client"` only when a component explicitly requires browser interactivity, standard React hooks, or Liveblocks multiplayer hooks (e.g., `useMutation`, `useStorage`, `useOthers`).
- Keep Next.js route handlers completely synchronous and fast. Any long-running AI generation or Markdown compilation MUST be offloaded to `Trigger.dev` background tasks.
- Mutate visual architecture state strictly via Liveblocks mutations to utilize CRDTs properly. Do not use local React state (`useState`) to manage node positions or connections, as this will cause state desynchronization.

## Styling

- Use the exact semantic custom property tokens defined in `ui-context.md`. Do not use raw Tailwind color classes (e.g., `bg-slate-800` or `#1F2937`).
- Reference color tokens directly via their Tailwind utility names: `bg-app`, `bg-surface`, `text-primary`, `text-secondary`, `accent-primary`, `accent-ai`, `accent-ai-glow`, etc.
- Maintain the custom border radius scale defined in the design system: `rounded-radius-sm` for small UI components, `rounded-radius-md` for standard diagram nodes and buttons, `rounded-radius-lg` for large structural elements like sidebars.

## API Route Structure

- Validate and parse all request inputs using Zod before executing any further logic.
- Enforce strict authentication checks utilizing Clerk.
- Validate project access before mutation: query the Prisma database to confirm the user is either the `ownerId` or exists in the `ProjectCollaborator` junction table.
- Return consistent, predictable JSON response shapes.
- Keep route handlers thin. Push complex logic into `/lib` shared modules or enqueue jobs to `Trigger.dev`.

## Data and Storage

- Project metadata, user profiles, and collaborator relationships belong strictly in PostgreSQL via Prisma.
- Live, high-frequency canvas coordinates and node states belong strictly in Liveblocks memory.
- Generated `.md` tech specs belong in Vercel Blob. The PostgreSQL database only stores the URL reference to the blob.

## File Organization

- `app/` — Next.js routing layer handling UI application shell and HTTP API endpoints.
- `components/canvas/` — The interactive visualization layer owning the React Flow integration and custom node rendering.
- `components/ui/` — Dumb, stateless, reusable UI components.
- `lib/db/` — Data access layer for the Prisma client and PostgreSQL CRUD operations.
- `lib/ai/` — The generation engine containing Vercel AI SDK configurations, LLM system prompts, and Zod graph schemas.
- `jobs/` — Async worker layer containing Trigger.dev background task definitions.
- `app/api/webhooks/` — Integration layer for external event listeners (e.g., keeping internal users synced with Clerk via Svix).
